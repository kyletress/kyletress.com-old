Sledsheet

Sledsheet is a timesheet analysis application and a database of skeleton results dating back to 1999. I built the app in Ruby on Rails, and I am responsible for its design, development, and maintenance. The source code is on GitHub and you can view the site at www.sledsheet.com.

History

In 2007 I co-created a small social network and satirical news site for the skeleton community called SliderGuide. I was primarily responsible for the visual design, content, and front-end development of that site. My cofounder Chris built the back-end with Ruby on Rails, which was my introduction to the framework. As a designer, I was impressed with the readability of Ruby. I decided to learn Rails so I could build some personal projects.

Sledsheet was my first production app. Initially built on Rails 2, I re-wrote it for Rails 3.1 and again for Rails 4, mostly as an exercise in improving my skills and learning the new features of each release.

The Problem

I specifically learned Rails to build this app. I wanted an easy way to analyze my training timesheets, which are typically delivered as PDF files. As a sled travels down the track, we trigger a series of 6 timing eyes from start to the finish. The PDF shows the split time recorded at each timing eye, as well as an athlete’s position at that eye. While useful, there is additional data that can be extracted using simple math. For instance, by subtracting the split times between two eyes, you can see how long it took to travel a particular distance. This is called the intermediate time, and it’s useful for determining where an athlete gained or lost time on the track. Another useful calculation is the difference between two athletes (one usually being the fastest sled of the heat).

What makes Sledsheet useful is that you don’t have to calculate these numbers manually, and the analysis isn’t limited to a specific day. It’s just as easy (and far more useful) to compare to a different day with similar weather conditions, or perhaps to the fastest run ever on that track.

Technology and Features

Sledsheet supports manual entry of timesheet information, or automated entry from the fibt.com website. Using a custom Nokogiri script I was able to import almost 10,000 runs spanning 10 competitive seasons in just a few minutes. As a run is saved, the intermediates are calculated and stored, and the differences are calculated on the fly. Rank is calculated at the database level with Postgres. Run data can be graphed for a visual representation of how each athlete performed compared to the rest of the field.

The official track PDF timesheet can be uploaded to Amazon Web Services, or the user can export a timesheet with specific athletes and data (not yet available on the current release).

The site is hosted on Heroku.

The Data Model

Sledsheet revolves around the Timesheet model. A Timesheet has_many entries. An Entry is a collection of runs on a given day for a particular Athlete. Each Entry has_many runs. A Run holds the actual split data for one trip down the track. There are additional models for things like Season, Track, and Circuit.

The Future

I’m currently building a JSON API with basic authentication to act as the backend to an iOS app. Presently my plan is to release the app purely as a database of historical race records, but in time I’d like to add a live timing feature. Ultimately, I think the analysis and management features are best suited to a mobile app rather than a web app, so I’ll probably build a second app for that.
